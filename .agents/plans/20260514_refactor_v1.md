# 重构方案：Siyuan Background Cover Plugin v1.0

## 目标

精简功能与代码，升级至最新思源 API，选择性对齐 VSCode 插件功能。

---

## 一、核心约束分析

### 1.1 VSCode vs 思源插件权限差异

| 能力 | VSCode 插件 | 思源插件 (desktop) | 思源插件 (mobile/browser/docker) |
|------|------------|-------------------|-------------------------------|
| 运行环境 | Node.js 进程 | Electron（nodeIntegration: true） | 纯原生 WebView，无 Node.js |
| 读本地文件 | `fs.readdirSync()` | `window.fs.readdirSync()` | 只能通过内核 HTTP API |
| 文件夹对话框 | `showOpenDialog()` | `<input webkitdirectory>`（无路径） | `<input webkitdirectory>`（无路径） |
| 本地路径渲染 | `fs` + DOM | `url("file:///...")` | 不支持 `file:///` |
| 写外部路径 | `fs.writeFileSync()` | `window.fs.writeFileSync()` | 只能通过 `/api/file/putFile` |

**`window.fs` — 关键突破**：

在思源开发者命令行中发现 `window.fs` 对象，这是 Electron 暴露的完整 Node.js `fs` 模块映射。Elrctron desktop 端拥有完整文件系统能力。

**移动端限制**：

Siyuan 移动端 (Android/iOS/HarmonyOS) 是纯原生 WebView，不存在 Node.js 运行时，`window.fs` 为 `undefined`，`file:///` 协议不可用。测试降级方式：

```typescript
if (!(window as any).fs) {
    // dev 模式下直接提示，生产环境隐藏该输出
    console.log("当前非桌面端，本地文件夹功能不可用");
    return;
}
```

### 1.2 源类型可用性矩阵

| 源类型 | desktop | mobile / browser / docker | 枚举方式 | 渲染方式 |
|--------|---------|--------------------------|---------|---------|
| `local` | **可用** | 不可用 | `window.fs.readdirSync()` | `url("file:///...")` |
| `upload` | **可用** | **可用** | `readDir("data/public/...")` | `url("/public/...")` 或 `file:///` |
| `assets` | **可用** | **可用** | `readDir("data/assets/...")` | `url("/assets/...")` |
| `url` | **可用** | **可用** | HTTP 下载 | `url(...)` |

**结论**：desktop 端支持全部三种源（`local` 提供零上传体验，同时保留 upload/assets 兼容）；非 Electron 端两种源可用（upload/assets）。URL 方式合并到 upload 源的添加入口中。

#### 2.3.1 assets 子文件夹引用实现要点

`data/assets/` 随着笔记增加可能包含数万文件，直接全量扫描不可行。改为让用户选取**特定子文件夹**。

```
工作原理：
  data/assets/                    ← 思源笔记所有附件存放处（可能数万文件）
    wallpaper/                    ← 用户手动创建，专门放壁纸
      sunset.png
      mountain.jpg
    travel-photos/                ← 用户选定作为图片源
      beach.png
      tokyo.jpg
    image-20230523085812-k3o9t32.png   ← 散落在根目录的大量笔记附件，不纳入
    foo-20210719092549-9j5y79r.png
    ...

  用户操作：
    1. 在设置界面打开「assets 文件夹选取器」
    2. 展开 data/assets/ 目录树，勾选 wallpaper/ 和 travel-photos/
    3. 支持嵌套子文件夹（可选递归扫描）
  
  插件行为：
    1. readDir("data/assets/wallpaper/") → 列出 wallpaper/ 下文件
    2. readDir("data/assets/travel-photos/") → 列出 travel-photos/ 下文件
    3. 按扩展名过滤图片/视频
    4. 直接使用 CSS url("/assets/wallpaper/sunset.png") 引用
    5. 不复制、不上传、零额外存储
```

**关键优势**：
- 用户笔记中已有的图片直接作为背景使用，避免一张图片占用双倍空间
- 每个选中的子文件夹作为一个图片源，可独立启用/禁用
- 不会加载整个 `assets/` 目录，避免性能问题

#### 2.3.2 图片源统一管理设计

三种源类型，desktop 端全支持，非 Electron 端排除 `local`：

```
desktop (Electron, window.fs 可用)          fallback (mobile/browser/docker)
  type='local'   ← ← 独占                    type='local'   ✗ 隐藏
  type='upload'  ← → 通用                    type='upload'  ← → 通用
  type='assets'  ← → 通用                    type='assets'  ← → 通用
```

**URL 图片/视频合并到 upload 源**：用户输入 URL 后，插件下载文件（desktop:`window.fs` / fallback:HTTP fetch），上传到 `upload` 源所在的 `data/public/` 目录。URL 下载得到的文件与其他上传文件在同一个随机池中无差别对待，不需要一个独立的 `url` 源类型。

```
sourceManager
├── sources: ImageSource[]            # 所有已注册的源
│
├── detectPlatform() → 'desktop' | 'fallback'

├── addSource(type, path, label)
│   ├── type='local'  → 用户输入本地文件夹绝对路径
│   ├── type='upload' → HTML <input> 文件选择器 / URL 下载入口
│   └── type='assets' → 文件夹树选取器 (readDir 遍历)
│
├── removeSource(id)

├── scanAll() → ImageItem[]          # 非 Electron 端自动跳过 local
│   ├── type='local'  → (desktop) fs.readdirSync(path) → 过滤扩展名
│   ├── type='upload' → readDir("data/public/...") → 过滤扩展名
│   └── type='assets' → readDir("data/assets/<子文件夹>") → 过滤扩展名
│
└── pickRandom(excludeUrl)            # 从统一列表中随机选取
```

**三种源的差异处理：**

| 维度 | local | upload | assets |
|------|--------|--------|----|
| 适用平台 | desktop only | 所有 | 所有 |
| 文件归属 | 用户自有目录，只读 | 插件目录，可增删 | 思源资产，只读 |
| 枚举方式 | `window.fs.readdirSync()` | `readDir()` | `readDir()` |
| 渲染 URL | `file:///` | `/public/...` 或 `file:///` | `/assets/...` |
| 云同步 | 不涉及 | 仅 fallback 端 | 文件会同步 |
| 存储开销 | 零 | 占用 data/ 空间 | 零 |

## 三、资源类型区分（图片 vs 视频）

### 3.1 渲染模式自动切换

`bgRender.ts` 根据文件扩展名决策渲染方式：

```
ImageItem
├── type='image' → <canvas> 元素
│   └── CSS background-image: url(...)
│       + background-size: cover
│       + opacity / filter: blur()
│       + background-position: X% Y%
│
└── type='video' → <video> 元素
    └── <video src="..." loop muted autoplay>
        + opacity / filter: blur()
        + object-fit: cover
        + 不支持 position 偏移（CSS 限制）
```

### 3.2 类型判定策略

```
判定优先级（快路径优先）:
1. 扩展名匹配已知列表（在设置中，可通过string自定义，并检查输入是否合法） → 直接分类
    图片: .png .jpg .jpeg .gif .webp .bmp .svg .avif
    视频: .mp4 .webm .ogg .mov .avi .mkv
2. 无法判定 → 跳过，不加入随机池（在console.log中记录一下哪些文件被掠过了）
```

**MIME 探测**：desktop 端可通过 `window.fs.readFileSync()` 读取文件头判断真实类型，但开销较高且收益有限。扩展名判定在 desktop 和 fallback 端均可用，性能更好。

### 3.3 统一池内混用

图片和视频共享同一个随机池：
- `scanAll()` 返回 `ImageItem[]`，每个 item 带有 `type: 'image' | 'video'`
- `bgRender` 渲染时根据 type 选择 canvas 或 video 元素
- 随机选择不区分类型，允许图片和视频混用
- 用户可通过 UI 过滤「仅图片」「仅视频」「全部」

### 3.4 视频特有处理

视频的 CSS 能力受限（video 元素不支持 `background-*` 属性）：

| 特性 | 图片 (canvas) | 视频 (video) |
|------|-------------|------------|
| opacity | body opacity | video CSS opacity |
| blur | canvas CSS filter | video CSS filter |
| X/Y 偏移 | `background-position: X% Y%` | **不支持** |
| 尺寸模式 | `background-size: cover` | `object-fit: cover` |
| 自动播放 | N/A | autoplay (需 muted) |
| 循环 | N/A | loop |
| 性能 | 静态，低开销 | 持续解码，较高 |

**典型用途**：静态壁纸用图片，动态氛围用视频（如飘雪、星空、流水），两者可由同一个随机池管理。

## 四、存储与同步方案

### 4.1 核心原则

文件资源不进入思源工作空间时，不会触发云同步。desktop 端 `local` 源直接引用外部目录，天然隔离。非 Electron 端 `upload` / `assets` 源仍在 `data/` 内，会同步但接受此行为。

### 4.2 全部配置本地化

```typescript
// setLocalStorageVal("siyuan-plugin-background-cover", { ... })
// 全部配置可 reset 时清除，替代旧版 configs.json 文件名语义
interface AppConfig {
  // 显示
  activate: boolean;
  opacity: number;            // 0.0 - 1.0
  blur: number;               // 0 - 10 (px)
  positionX: number;          // 0 - 100 (%) 全局默认
  positionY: number;

  // 图片源
  localFolders: string[];     // desktop: 外部文件夹绝对路径
  assetDirs: string[];        // data/assets/ 下用户勾选的子目录路径

  // upload 源不持久化 — 固定路径 data/public/siyuan-plugin-bgcover/，启动时直接 readDir 扫描

  // 自动刷新
  autoRefresh: boolean;
  autoRefreshTime: number;    // 分钟

  // 主题
  disabledThemes: { dark: string[]; light: string[] };

  // 每图片偏移覆盖（key = 完整渲染 URL，用 URL 而非 hash 因为不需要跨设备同步）
  imageOverrides: Record<string, {
    positionX?: number;
    positionY?: number;
  }>;

  // 运行时状态
  currentFile: string | null;
}
```

**没有 `ImageSource` 接口**。upload 固定路径不需要持久化；local/assets 只是字符串数组，不再需要 `id`、`type`、`label`、`enabled` 等字段。

### 4.3 sourceManager 扫描逻辑

```
scanAll() 启动时：
1. upload → 固定路径 /public/siyuan-plugin-bgcover/ → readDir → 过滤扩展名
2. assetDirs → 遍历配置中的每个子目录 → readDir → 过滤扩展名 → 路径不可访问则跳过
3. localFolders → desktop only, fs.readdirSync → 过滤扩展名 → 路径不可访问则跳过
→ 合并为统一 ImageItem[]

文件增删时增量更新：
- 用户上传 / 删除文件 → 仅重新扫描 upload 目录
- 用户添加 / 移除 source → 重新扫描该 source 目录
```

### 4.4 per-image 偏移覆盖

key 使用完整渲染 URL（`file:///home/.../img.jpg` 或 `/assets/sub/img.png` 或 `/public/.../hash.jpg`）：

- 随机选中图片时 → 查 `imageOverrides[url]`
- 找到 → 用覆盖的 positionX/Y 渲染
- 未找到 → 用全局默认 positionX/Y

不考虑跨设备迁移：系统重装后视为新设备重新配置。文件夹被移动后文件夹源入口在 UI 中显示不可访问，用户删除重建即可。不需要 hash 多一层间接。

### 4.5 为何移除 `sizeMode`

`cover` 即可满足绝大多数需求：
- 图片自动缩放填满屏幕，不变形（保持宽高比）
- `repeat` 大多重复背景视觉效果奇怪
- `center`/`contain` 边缘容易留白

不提供选择，简化代码和 UI。

### 4.6 为何不需要 `prevTheme`

- 背景层是独立的 canvas/video 元素，不受主题 CSS 影响
- 主题变化时只需检查 `disabledThemes` 列表：如果新主题在禁用列表中，隐藏背景；否则显示
- 用 `MutationObserver` 监听 `<html>` 属性变化实时检测，无需持久化

### 4.7 文件资源同步

| 源类型 | 文件位置 | 云同步 | 说明 |
|--------|---------|--------|------|
| `local` | 用户指定绝对路径 | 不涉及 | 完全在思源工作空间之外 |
| `assets` | `data/assets/<子文件夹>` | 会同步 | 本来就是笔记附件，同步合理 |
| `upload` | `data/public/siyuan-plugin-bgcover/` | 会同步 | 含 URL 下载的图片，可手动加 syncignore |

### 4.8 API 迁移

| 废弃/旧 API | 替代方案 |
|------------|---------|
| `/api/storage/setLocalStorage` (整文件覆写) | `/api/storage/setLocalStorageVal()` (PR #17482) |
| `/api/storage/getLocalStorage` | `/api/storage/getLocalStorageVal()` |
| `plugin.saveData()` / `plugin.loadData()` | 不再需要，全部走 `localStorage` |
| `ts-md5` 分片哈希去重 | 文件名去重 (upload) / 无需去重 (local) |
| `prevTheme` 持久化检测 | `MutationObserver` 实时检测 |
| `sizeMode` 多模式切换 | 统一 `cover` |

## 六、目标代码架构

### 设计约定

| 目录 | 职责 | 原则 |
|------|------|------|
| `src/` 根 | 入口、类型、常量 | 仅保留全局共用的顶层文件 |
| `libs/` | 框架级基础件 | 通用封装，不依赖本插件业务逻辑（SettingUtils、dialog） |
| `utils/` | 业务领域工具 | 依赖本插件业务上下文（api、theme） |
| `stores/` | 状态管理 | 单例 store，管理配置和运行时状态 |
| `services/` | 核心服务 | 背景渲染引擎、图片源管理 |
| `ui/` | Svelte 组件 | 所有视图层，含组件内联的对话框调用 |

### 目录结构


#### i18n 机制

| 环节 | 说明 |
|------|------|
| 源文件 | `public/i18n/<lang>.json`，扁平 JSON 键值对 |
| 构建 | Vite 自动将 `public/` 内容输出到 `dist/`（`i18n/` 子目录） |
| 运行时 | 思源自动读 `i18n/` 目录注入 `this.i18n`，无需手动加载 |
| 代码中使用 | `this.i18n.keyName` 直接访问，支持 `${var}` 模板插值 |
| 命令快捷键显示 | `addCommand({ langKey: "selectManual", hotkey: "⇧⌘F6" })` |
| plugin.json | `displayName` / `description` / `readme` 字段支持 `{ "default": "...", "zh_CN": "..." }` |

## 七、执行阶段

可以参考旧代码，但不受其限制，重新设计实现细节和代码结构。每个阶段完成后都要进行测试验证，确保功能正确且性能可接受。

### Phase 1：项目基建
- 搭建 Vite + Svelte 5 项目骨架
- 配置 `vite.config.ts`（lib mode, CJS, external siyuan）
- 安装并配置 `vitest` + `@testing-library/svelte` + `jsdom`
- 创建 `tests/` 目录结构和 `vitest.setup.ts` 全局 mock
- 更新 `plugin.json` (minAppVersion, backends, frontends)
- 更新 `package.json` (dependencies, scripts)
- 迁移 plugin 入口文件 `index.ts`

### Phase 2：存储改造
- 移除 `/api/storage/setLocalStorage` / `getLocalStorage` 调用 (Issue #37)
- 全部配置迁移到 `local.json` 单层存储：`setLocalStorageVal("siyuan-plugin-background-cover", {...})`
- 使用 PR #17482 新增的 `getLocalStorageVal` / `setLocalStorageVal` / `removeLocalStorageVal` API
- 移除 `plugin.saveData()` / `plugin.loadData()` 依赖，不再写入 `data/storage/petal/`
- 移除 ts-md5 依赖，实现文件名去重
- 修改重置配置的函数，有能力清除之前旧版本的 `local.json` 和 `data/storage/petal/{plugin}` 和 `data/public/{plugin}` 中的相关数据和文件

### Phase 3：功能精简
- 移除 Live2D 桩代码、Bug 报告、Android 提示、Demo 图片、旧版迁移逻辑
- 清理未使用的导入和注释代码
- 合并重复的工具函数

### Phase 4：图片源管理
- 实现 `utils/fs.ts` — `window.fs` 安全封装（`isDesktop()` 检测 + 降级 shim）
- 实现 `sourceManager.ts` — 双路径图片源管理器
- 三种源类型：`local`(desktop直读) / `upload`(通用，含URL下载入孔) / `assets`(通用)
- 非 Electron 端自动过滤 `local` 源
- 多源混合统一随机池

### Phase 5：背景渲染重写
- 重写 `bgRender.ts`，支持图片 + 视频双模式
- 统一使用 `cover` 尺寸模式（不提供选择）
- 保留透明度、模糊、位置偏移控制
- 视频特性：loop, muted, autoplay

### Phase 6：UI 重写
- 用 Svelte 5 组件替代所有 HTML 模板字符串
- 设置面板：配置 tab + 资源池 tab（含文件树、源管理、鼠标悬浮预览）+ 主题 tab + 关于 tab
- 资源池 tab：三种源类型（upload/assets/local）统一树形展示，按 §4.9 设计
- 顶部栏菜单：按 §4.9 层级设计
- 鼠标悬浮缩略图预览

### Phase 7：收尾
- 清理依赖，更新 README
- 全量运行单元测试 + 组件测试 (`pnpm test`)
